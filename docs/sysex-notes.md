## Primary spec

**Official Roland "MD-500 MIDI Implementation" v1.00, dated 2017-07-01**
(`docs/spec/MD-500_MIDI_Imple_eng01_W.pdf`, downloaded from
`static.roland.com/assets/media/pdf/MD-500_MIDI_Imple_eng01_W.pdf`; extracted to
`docs/spec/midi_imple.txt`). The owner's manual and editor manual are also
extracted under `docs/spec/` for parameter semantics.

**Status: device-verified on 2026-06-12** against a real MD-500 (USB) ‚Äî see
[On-device verification](#on-device-verification). Framing, identity, the address
map, multi-chunk dump reassembly, and every area's decode are confirmed on
hardware. Writes (`sync`) are not yet exercised.

## Framing (from spec ¬ß1)

| Field | Value |
|---|---|
| SOX | `F0` |
| Manufacturer | `41` (Roland) |
| Device ID | `00`‚Äì`1F` (UI 1‚Äì32); `7F` = broadcast |
| Model ID | `00 00 00 43` (**4 bytes** ‚Äî note RE-202 used a 5-byte id) |
| RQ1 (read) | `11`, then 4-byte address + 4-byte size |
| DT1 (write) | `12`, then 4-byte address + data |
| Checksum | Roland standard `(128 ‚àí (sum(addr+data) mod 128)) mod 128` |
| EOX | `F7` |

Addresses and the RQ1 size field are **four 7-bit bytes** (a 28-bit value, carry
at `0x80`). Verified internally against the spec anchor BANK 99 =
`30 00 00 00` + 99¬∑`00 04 00 00` = `33 0C 00 00`.

### Identity (from spec ¬ß2)

- Request: `F0 7E <dev> 06 01 F7`.
- Reply: `F0 7E <dev> 06 02 41 43 03 00 00 ss ss ss ss F7` ‚Äî family code `43 03`,
  family number `00 00`, then a 4-byte software revision.

## Address map (spec ¬ß3)

| Address | Area | CLI area | Bytes |
|---|---|---|---|
| `00 00 00 00` | Setup Common (current patch) | `setup` | 3 |
| `10 00 00 00` | System Common | `system` (block 1) | 34 |
| `10 00 10 00` | System Control | `system` (block 2) | 16 |
| `20 00 00 00` | Program Change map (384 √ó 3-byte patch refs) | `pc-map` | 1152 |
| `30 00 00 00` | BANK Temporary ‚Äî SIMUL params | `bank` | 4 |
| `30 00 10 00` | BANK Temporary ‚Äî PATCH A | `patch-a` | 586 |
| `30 00 20 00` | BANK Temporary ‚Äî PATCH B | `patch-b` | 586 |
| `30 00 30 00` | BANK Temporary ‚Äî PATCH C | `patch-c` | 586 |
| `30 04 00 00` ‚Ä¶ `33 0C 00 00` | BANK 01‚Ä¶99 (stride `00 04 00 00`) | wasm only | ‚Äî |

Within any bank: SIMUL params `+00 00 00`, PATCH A `+00 10 00`, B `+00 20 00`,
C `+00 30 00`. The CLI exposes the global areas + the live Temporary bank; the
99 stored banks are reachable via the wasm layer (`bankBase`, `patchBase`).

## Patch block (586 bytes) ‚Äî layout

`0x000` name (16 ASCII) ¬∑ `0x010` common header (mode + LFO/EQ/output) ¬∑
`0x023` twelve modulation modes laid flat (CHORUS, FLANGER, PHASER, C-VIBE,
VIBRATO, TREMOLO, DIMENSION, RING MOD, ROTARY, FILTER, SLICER, OVERTONE) ¬∑
`0x173` per-patch footswitch/expression control ¬∑ `0x180` eight 25-byte ASSIGN
slots ¬∑ `0x248` Assign Input Sens ¬∑ `0x249` LED status. Every offset is present
regardless of the active mode. See `md500-core/src/patch.rs` for the field-level
map; the offset checklist is `docs/spec/patch_offsets.txt` (generated by
`docs/spec/offsets.py`).

### Modelling notes / open questions

- Multibyte values are **nibble-packed**, MSN first (Rate 3B, BPM 5B, pre-delay
  3B, rotor/horn 2B, ring-mod & rotary rates 3B, exp targets 4B, assign
  target 3B / target min/max 6B). All other params are single 7-bit bytes.
- The mode sub-type selectors are all proper enums now, recovered by reading the
  rendered PDF (the `pdftotext` dump interleaves the two columns, which is why an
  earlier pass left four as raw `u8`): PHASER `prime_g/prime_b/script`, VIBRATO
  `prime/scanner`, TREMOLO `prime_t/prime_p/twin/deluxe`, DIMENSION `m1..m4/user`.
- The huge ASSIGN **source** (`0..=69`) and **target** (`0..=264`) lists, and the
  scaled `target_min`/`target_max` values, are modelled as integers (not enums).
  These index large name lists; the intended home for those names is the kit's
  `Catalogs` name‚Üînumber tables (like CK's voices/effects) ‚Äî extract the lists
  from the spec (`assign_source` / `assign_target`) and tag the `source`/`target`
  ParamMeta with a `catalog:` hint, leaving the wire values as integers. Not yet
  done. `target_min`/`target_max` are device-internal scaled values and stay
  numeric.
- `Dummy` bytes in the spec (patch `0x74`, `0x89`, `0x8F`, control `0x174`) are
  preserved as `reserved_*` fields so decode‚Üíencode is byte-exact.

## On-device verification

**Performed 2026-06-12** against a real MD-500 over USB (port "MD-500"). Results:

- **Device id is `0x10`** (UI "17", the factory default) ‚Äî *outside* the generic
  engine's `--device 0..=15` range. A channel-derived RQ1 (device id `0x00`) got
  **no reply**; addressing requests to **broadcast `0x7F`** works (the reply comes
  back tagged with the device's own `0x10`). The crate now always broadcasts (see
  `device_id` in `device.rs`).
- **Identity**: reply `F0 7E 10 06 02 41 43 03 00 00 00 00 00 01 F7` ‚Äî family code
  `43 03`, software rev `‚Ä¶01`. `md500 identity` ‚Üí "MD-500 detected". ‚úì
- **Large regions are split into multiple DT1 chunks at successive addresses.**
  A patch dump returned **242 + 242 + 102 = 586 bytes** at `30 00 10 00`,
  `30 00 11 72`, `30 00 13 64`. The decoder reassembles by chaining contiguous
  DT1 addresses (`collect_block`). The PC map (1152 B) reassembles the same way.
- **All areas decode on hardware**: `setup` (‚Üí 07A), `system` (two blocks),
  `bank`, `pc-map` (384 entries, default map 01A/01B/01C/02A‚Ä¶), and `patch-a`
  (name "DIMENSION 421 ST", mode `dimension` `m2`). `lint` re-encodes the dumped
  patch through the typed model cleanly.
- **Writes need the same chunking.** A single 586-byte DT1 is mis-parsed ‚Äî a
  no-op rewrite corrupted one byte (`effect_level` 32‚Üí31). Splitting the write
  into ‚â§242-byte DT1 frames at successive addresses (`push_dt1` ‚Üí
  [`MAX_DT1_DATA`]) makes `md500 sync patch-a --verify` round-trip and read back
  byte-identical to the original. ‚úì Writing the Temporary patch is immediate and
  non-destructive to other patches.
